## 1. 范围与目标

- 范围：仅验证 LAN 双机稳定性与重放正确性，功能冻结（不新增业务功能）。
- 目标：验证 Host->Edge 在弱网与长时间运行下可恢复、可观测、可回归，并完成 mTLS 升级演练。
- 对应 Gate 条件：
  - 24h soak test 无致命中断。
  - Capture 丢失率 <= 0.2%。
  - 重放一致性校验通过率 = 100%（同一 `capture_id` 结果一致）。
  - mTLS 握手与证书轮换演练通过率 = 100%。
  - UI 关键路径在 LAN 24h 中致命中断次数 = 0。
  - 观测 KPI（non-blocking）：Chat 引用覆盖率目标 >= 92%，Stretch >= 95%。

## 2. 环境与输入

- 运行环境：
  - Host：macOS（Client）
  - Edge：macOS（Server）
  - 网络：同一 LAN，含可控弱网注入（丢包/抖动/瞬断）
- 配置与数据集：
  - 24h soak 数据流（正常工作负载 + 峰值负载）
  - 故障注入场景：断网、重复包、乱序、短时 Edge 重启、证书轮换
  - UI 关键路径脚本：`timeline -> search -> chat -> citation -> frame`
- 依赖版本：
  - `myrecall_client.env` / `myrecall_server.env` 版本
  - mTLS 证书与轮换脚本版本
  - 验收脚本/统计脚本版本

### 2.1 指标口径与样本说明（必填）

- 口径基线版本（默认 `MyRecall/docs/v3/gate_baseline.md`）：v1.4
- 指标样本数：
  - Capture（丢失率/重放一致性）：>= 200 captures（建议 >= 1000）
  - Search：>= 200 queries
  - Chat 请求成功率：>= 100 requests
  - Citation Coverage（Soft KPI）：>= 100 个问答样本
- 统计时间窗：
  - 主窗：连续 24h soak
  - 故障注入：每类场景至少 3 轮
- 百分位算法：Nearest-rank（每轮剔除前 10 个预热样本）

## 3. 验收步骤

1. 部署 Host/Edge 双机并完成连通性检查，确认 `/v1/*` 路径可用。
2. 启动 24h soak，持续产生 capture/search/chat 流量并采集全量日志。
3. 在 soak 期间执行弱网注入（丢包/抖动/瞬断），观测队列、重试与恢复行为。
4. 对同一批 `capture_id` 执行重复发送与乱序重放，验证幂等与结果一致性。
5. 执行 mTLS 握手验证与证书轮换演练（含失效证书/过期证书场景）。
6. 统计 Capture 丢失率、重放一致性、Search/Chat 延迟分位并产出指标表。
7. 运行 UI 关键路径脚本，记录致命中断次数与失败原因。
8. 统计 Chat 引用覆盖率（non-blocking），若低于目标必须附整改动作与回归计划。
9. 汇总证据：原始日志路径、SQL/统计脚本、指标汇总表（P50/P95/P99）。
10. 输出 Gate 结论（Pass/Fail）与阻塞项。
