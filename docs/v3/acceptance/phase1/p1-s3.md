## 0. screenpipe 对照与对齐

- screenpipe 做法：AX-first，OCR-fallback；对特定应用可配置 OCR 偏好；仅存储原始文本，不做索引时 AI 增强。
- 本阶段对齐：完全对齐 AX-first/OCR-fallback 行为与存储策略；以 Edge 作为唯一重处理节点；不生成 caption/keywords/fusion_text，且不写入 `ocr_text_embeddings`（014A/015A）。
- 口径约束（Scheme C SSOT）：AX 成功帧写 `accessibility`（不写 `ocr_text`），OCR fallback 帧写 `ocr_text`（不写 `accessibility`）；验收与 SQL 对账以 `data-model.md` §3.0.3 写入语义为准。

## 1. 范围与目标

- 范围：Edge 处理链路（AX-first + OCR-fallback）、按 `frames.text_source` 分表写入原始文本（AX→`accessibility`，OCR→`ocr_text`）、可追溯日志。
- 目标：验证决策正确性、可追溯性与 UI 上的处理来源可解释性。
- 对应 Gate 条件：
  - AX-first/OCR-fallback 决策日志可追溯率 >= 95%。
  - 索引时零 AI 增强检查通过率 = 100%（caption/keywords/fusion_text/embedding 写入为 0）。
  - 处理来源字段 UI 展示完整率 = 100%。

## 2. 环境与输入

- 运行环境：同 P1-S1。
- 配置与数据集：
  - 普通应用样本（AX 可用）
  - `ocr_preferred_apps` 样本（强制 OCR）
  - AX 不可用样本（触发 fallback）
- 依赖版本：OCR 引擎与处理 pipeline 版本

### 2.1 指标口径与样本说明（必填）

- 口径基线版本（默认 `MyRecall/docs/v3/gate_baseline.md`）：v1.1
- 指标样本数：
  - 处理链路样本：>= 100 条处理记录（覆盖 AX 可用/OCR 偏好/AX 缺失三组）
  - 异常注入样本：每类异常（OCR 超时/AX 为空）至少 3 轮
- 统计时间窗：
  - 主窗：单次完整处理验收窗口
  - 异常窗：每类异常独立记录起止时间
- 百分位算法：Nearest-rank（若统计百分位，剔除前 10 个预热样本）

## 3. 验收步骤

1. 准备三组样本数据：AX 可用、OCR 偏好、AX 缺失。
2. 触发处理流程并记录每条 capture 的决策路径（AX/OCR/fallback 原因）。
3. 检查分表写入与 `frames.text_source` 一致：
   - `text_source='ocr'` 的完成帧应有 `ocr_text` 记录；
   - `text_source='accessibility'` 的完成帧应有 `accessibility(frame_id)` 记录。
4. 用 SQL 与处理日志核对索引时增强禁用项：`caption/keywords/fusion_text` 写入为 0，`ocr_text_embeddings` 新增记录为 0。
5. 抽样核验至少 100 条处理记录，计算可追溯率。
6. 在 frame 详情页检查处理来源字段与处理时间戳展示完整性。
7. 注入处理异常（OCR 超时/AX 为空）并验证错误码、重试与降级日志。
