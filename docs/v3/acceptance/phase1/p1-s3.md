## 0. screenpipe 对照与对齐

- screenpipe 做法：vision-only 处理链在可用时生成 OCR/raw text，并避免索引时 AI 增强。
- 本阶段对齐：v3 主线只实现 OCR processing，不实现 AX/dual-path 决策逻辑，不把 `accessibility` 表作为活跃写入目标。
- P1 OCR 引擎策略：RapidOCR（single-engine policy），本阶段不做多引擎切换或对比验收。
- 零 AI 增强（OQ-014/OQ-015）：P1 不生成 `caption/keywords/fusion_text`，不写入 `ocr_text_embeddings`；本阶段做防守性确认。

## 1. 范围与目标

- 范围：Edge OCR 处理链路、`ocr_text` 持久化、`frames.text_source='ocr'`、失败帧 `failed` 语义、处理来源可解释性、**UI 处理来源展示**
- **UI 实现要求**：P1-S3 必须完成 Grid `/` 的 frame 卡片展示（详见 §3.9）：
  - Card Header：显示 `app_name`、`window_name`、`device_name`

  - 状态区分：`pending`/`processing`/`completed`/`failed` 视觉可区分
  - ⚠️ **此为实现要求，非可选功能，不可跳过**
- 目标：验证 OCR-only 处理主线正确、可追溯、可恢复，**且处理结果在 UI 可见**

  - 索引时零 AI 增强检查通过率 = 100%
  - 处理来源字段 UI 展示完整率 = 100%（硬性交付物，不可跳过）

### 1.2 实现策略：独立 v3 Worker

> **背景**：现有 `openrecall/server/worker.py` 包含 caption/keywords/fusion_text/embedding 等 AI 增强逻辑，与 P1-S3 承诺的"OCR-only + 零 AI 增强"严重偏离。为避免破坏现有功能并保证阶段隔离，需创建独立的 v3 处理模块。

#### 1.2.1 架构决策

- **新建模块**：`openrecall/server/processing/v3_worker.py`（独立文件，不复用旧 worker）
- **原因**：
  1. 现有 worker 包含 `ai_provider.analyze_image()`、`keyword_extractor.extract()`、`build_fusion_text()` 等与 OQ-014/OQ-015 冲突的逻辑
  2. 现有 worker 写入 FTS 文档（`sql_store.add_document()`），而非 `ocr_text` 表结构
  3. 通过独立模块实现 `processing_mode=noop/ocr` 切换，避免条件分支污染旧代码
- **废弃策略**：v3 worker 成熟后，旧 worker 可标记为 deprecated，P2+ 统一清理

#### 1.2.2 v3_worker.py 核心职责

```python
# 伪代码结构示意
class V3ProcessingWorker(threading.Thread):
    """
    P1-S3+ OCR-only 处理 Worker。

    与旧 worker.py 的关键区别：
    - 不初始化 vision_provider / embedding_provider / keyword_extractor
    - 只调用 RapidOCRBackend.extract_text()
    - 写入 ocr_text 表（非 FTS 文档）
    - 设置 frames.text_source='ocr'
    - 支持 processing_mode='noop' 模式（P1-S1 兼容）
    """

    def run(self):
        while not self._stop_event.is_set():
            if self.processing_mode == "noop":
                # P1-S1: 仅驱动状态机，不执行 OCR
                self._advance_queue_state()
            else:  # processing_mode == "ocr"
                # P1-S3+: 执行 OCR 处理
                self._process_with_ocr()

    def _process_with_ocr(self, task):
        # 1. capture_trigger 校验（fail-loud）
        # 2. OCR 执行（RapidOCRBackend）
        # 3. 写入 ocr_text 表
        # 4. 设置 frames.text_source='ocr'
        # 5. 幂等防御（两层检查）
        pass
```

#### 1.2.3 processing_mode 切换机制

| 阶段 | `processing_mode` 值 | Worker 行为 |
|------|---------------------|-------------|
| P1-S1 | `"noop"` | 仅推进队列状态机（pending → processing → completed），不执行 OCR |
| P1-S3+ | `"ocr"` | 执行 OCR 并写入 `ocr_text` 表 |
| 其他 | 任意其他值 | 旧 ProcessingWorker（兼容保留，v3 主线不使用） |

- **配置入口**：`/v1/ingest/queue/status` 响应中的 `processing_mode` 字段
- **启动日志**：`MRV3 processing_mode={noop|ocr}`（与 spec.md §4.7.1 对齐）
- **切换时机**：阶段验收通过后，由配置或代码版本切换
- **环境变量**：`OPENRECALL_PROCESSING_MODE=noop|ocr`

```python
# openrecall/server/__main__.py

def _start_noop_mode():
    from openrecall.server.queue_driver import NoopQueueDriver
    driver = NoopQueueDriver()
    driver.start()
    return driver

def _start_ocr_mode():
    """启动 P1-S3+ OCR-only 处理 Worker。"""
    from openrecall.server.processing.v3_worker import V3ProcessingWorker
    worker = V3ProcessingWorker(processing_mode="ocr")
    worker.start()
    return worker

def _preload_ocr_model():
    """仅加载 OCR 模型（OCR-only 模式专用）。"""
    from openrecall.server.ocr.rapid_backend import RapidOCRBackend
    _ = RapidOCRBackend()  # 触发模型加载

def main():
    # ... 前置初始化 ...
    processing_mode = settings.processing_mode.strip().lower()
    worker = None

    if processing_mode == "noop":
        worker = _start_noop_mode()
    elif processing_mode == "ocr":
        _preload_ocr_model()      # 仅加载 RapidOCR
        worker = _start_ocr_mode()
        logger.info("MRV3 processing_mode=ocr")
    else:
        # 兼容：旧模式（v3 主线不使用）
        preload_ai_models()
        init_background_worker(app)

    # ... 后续逻辑 ...
```

| `error_message` | str | `frames.error_message` | 错误信息（失败时） |

#### 1.4.3 字段获取方式

```sql
-- Grid 列表查询（伪代码，供实现参考）
SELECT
    f.id AS frame_id,
    f.timestamp,
    f.app_name,
    f.window_name,
    f.device_name,
    f.capture_trigger,
    f.status,
    f.text_source,
    f.processed_at,
    f.error_message,
    o.text_length,
    o.ocr_engine,
    SUBSTR(o.text, 1, 100) AS ocr_text_preview
FROM frames f
LEFT JOIN ocr_text o ON o.frame_id = f.id
ORDER BY f.timestamp DESC
LIMIT ?;
```

### 2.1 S2b -> S3 数据流契约（Entry Contract，强制）

- Ownership 边界：S2b 只负责 capture completion；S3 独占 OCR 执行、`ocr_text` 写入、`frames.text_source='ocr'` 标记与失败语义。
- 输入契约：S3 消费截图、`capture_trigger`、`device_name`、`focused_context`（即 `app_name` + `window_name`）、`event_ts`、`timestamp` 等 capture-completion 产物；不得要求 `accessibility_text` / `content_hash` 作为主线前提。
- `capture_trigger` 契约：
  - 合法值：凡由 S2b 产出的 capture，`capture_trigger` 必须为非空且属于 P1 枚举（`idle` / `app_switch` / `manual` / `click`）
  - **非法值处理流程**（fail-loud）：
    1. OCR 执行前进行 `capture_trigger` 校验
    2. 若发现 `NULL` 或非法值，立即设置 `frames.status='failed'`
    3. 设置 `frames.error_message='INVALID_TRIGGER: {actual_value}'`
    4. 输出结构化日志：`MRV3 ocr_failed frame_id=X reason=invalid_trigger value=Y`
    5. **不得执行 OCR**
- 上下文边界：S3 直接消费 S2b 已冻结的 `app_name/window_name/device_name`；`browser_url` 在 P1 保持 reserved/NULL，不得在处理阶段补采或覆盖。
  - **验证方法**：
    - 代码审查：确认 processing worker 不调用 `get_active_window()` 等补采 API
    - 验收 SQL：`SELECT COUNT(*) FROM frames WHERE processed_at IS NOT NULL AND (app_name NOT IN (SELECT app_name FROM frames WHERE id = frames.id) OR window_name NOT IN (SELECT window_name FROM frames WHERE id = frames.id))` 应为 0
- 写入契约：
  - OCR 成功 -> 必须存在 `ocr_text(frame_id=frames.id)`，且 `frames.text_source='ocr'`
  - OCR 失败 -> `frames.status='failed'`，不得伪造 `text_source='accessibility'` 或 accessibility child rows
- v4 seam 边界：`accessibility` / `accessibility_fts` / `frames.accessibility_text` 为 schema seam，不参与本阶段 Gate。

**Simhash 丢弃帧说明**：
- S2b+（可选增强阶段）的相似帧丢弃发生在 **Host spool 层**，被丢弃的帧：
  - **不会**产生 `capture_id`
  - **不会**进入 Edge `/v1/ingest` 接口
  - **不会**出现在 `frames` 表中
  - **无需** S3 处理
- S3 的输入范围：仅处理已通过 simhash 过滤并成功 ingest 的帧（`frames.status='pending'`）

**OCR 失败处理策略（不重试）**：
- P1-S3 OCR 失败**不自动重试**，直接标记为 `failed`
- 理由：OCR 失败通常是图像损坏、格式不支持等确定性错误，重试成功率低
- 失败处理流程：
  1. OCR 执行失败（异常或返回空文本）
  2. 设置 `frames.status='failed'`
  3. 设置 `frames.error_message='OCR_FAILED: {具体原因}'`
  4. 输出结构化日志：`MRV3 ocr_failed frame_id=X reason={error_type}`
  5. **不写入 `ocr_text` 表**
- 幂等防御（防止重复处理同一帧）：
  - **处理前检查**：查询 `frames` 表，若 `status` 已为 `completed` 或 `failed`，直接跳过
  - **写入时检查**：写入 `ocr_text` 前再次检查是否已存在记录，若存在则拒绝写入并记录告警
  - 目的：避免并发场景下的重复处理，而非重试机制

## 3. 验收步骤

1. 准备 OCR 成功与 OCR 失败两组样本数据。
2. 触发处理流程并记录每条 capture 的 OCR 执行结果、错误码与完成时间。
3. 用 SQL 校验 OCR 成功帧写入正确性：
   ```sql
   SELECT COUNT(*) AS missing_ocr
   FROM frames f
   LEFT JOIN ocr_text ot ON ot.frame_id = f.id
   WHERE f.status='completed'
     AND f.text_source='ocr'
     AND ot.id IS NULL;
   ```
   判定规则：`missing_ocr = 0`。
4. 用 SQL 校验失败语义：
   ```sql
   SELECT COUNT(*) AS failed_with_ocr_row
   FROM frames f
   JOIN ocr_text ot ON ot.frame_id = f.id
   WHERE f.status='failed';
   ```
   判定规则：`failed_with_ocr_row = 0`。
5. 用 SQL 校验 v4 seam 未被误写：
   ```sql
   SELECT COUNT(*) AS unexpected_accessibility_rows
   FROM accessibility;
   ```
   判定规则：`unexpected_accessibility_rows = 0`。
   > **检查目的**：验证 v4 reserved seam (`accessibility` 表) 未被 v3 代码意外写入。OCR-only 主线下该值应为 0。
6. 确认零 AI 增强（防守性检查）：
   ```sql
    -- 确认 ocr_text_embeddings 表不存在（OQ-015，P1 不建，防守性检查）
   SELECT COUNT(*) AS table_exists
   FROM sqlite_master
   WHERE type='table' AND name='ocr_text_embeddings';
   ```
   判定规则：`table_exists = 0`。
   > **检查目的**：防守性验证，确保未提前实现 embedding 表（OQ-014/OQ-015）。
7. 抽样核验至少 100 条处理记录，计算 OCR 处理来源可追溯率。
8. 边界用例核验（每类至少 3 条样本）：
   - OCR 成功后必须写 `ocr_text` 并标记 `text_source='ocr'`
   - OCR 失败后必须进入 `failed`
   - OCR 处理不得覆盖 S2b 已冻结的 `device_name` / `focused_context`
9. 在 Grid `/` 的 frame 卡片中验证处理信息展示完整性。

    > **数据源**：参见 §1.4 UI 数据源定义。验收前需确认后端 API 已返回所需字段。

### 7.2 S3 -> S4 数据流契约（Exit Contract）

- Ownership 边界：S3 独占 OCR 执行与 `ocr_text` 写入；S4 仅消费 `ocr_text`、`frames.text_source` 和 `ocr_text_fts`。
- **数据一致性不变量**：
  - `ocr_text` 行数 MUST EQUAL `frames.status='completed' AND frames.text_source='ocr'` 行数
  - `ocr_text_fts` 索引行数 MUST EQUAL `ocr_text` 行数（FTS 触发器自动维护）
- **FTS 索引就绪**：OCR 成功后，FTS 触发器自动写入 `ocr_text_fts`；S4 搜索依赖该索引就绪
- S4 Entry 前置条件：S3 必须产出至少 1 条 `text_source='ocr'` 的 completed 帧（空库场景 S4 无法验证搜索能力）
