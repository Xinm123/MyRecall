## 0. screenpipe 对照与对齐

- screenpipe 做法：UI 通过后端 API 驱动核心流程（search/chat/timeline），并要求健康状态可观测。
- 本阶段对齐：对齐“API 可驱动 + 状态可观测”；不要求与 screenpipe UI 技术栈一致。

## 1. 范围与目标

- 范围：Host 上传链路（`POST /v1/ingest` + 幂等重试）、Edge ingest/queue 骨架（`GET /v1/ingest/queue/status`）、Edge 页面基础可用。
- 图片契约基线：主采集/主读取链路 JPEG（`/v1/ingest` 主契约 `image/jpeg`，`/v1/frames/:frame_id` 返回 `image/jpeg`）。
- 协议基线（019A）：单次幂等上传，`capture_id` 去重，`200 already_exists` 幂等语义，queue/status 端点可观测。
- 目标：在本机双进程下跑通"采集文件 -> 上传 -> 入队 -> 可观测"最小闭环，并验证 UI 基线可用。
- 对应 Gate 条件：
  - 同机断网恢复后可自动重传，且重复上传不重复入库（`200 already_exists`，frames 表无重复行）。
  - ingest 队列可观测（`GET /v1/ingest/queue/status` 返回 pending/processing/completed/failed）完整。
  - 图片格式契约一致性通过：主链路 JPEG；兼容输入若启用，需验证入库前转码为 JPEG。
  - API 命名空间一致性通过：对外仅 `/v1/*`；旧命名空间废弃回归仅覆盖 legacy 端点：`POST /api/upload`、`GET /api/search`、`GET /api/queue/status`、`GET /api/health`，并返回 301 重定向至对应 `/v1/*` + 记录 `[DEPRECATED]` 日志。
  - UI 基线路由可达率 = 100%。
  - UI 健康态/错误态展示检查通过率 = 100%。

## 3. 验收步骤

1. 启动 Edge 与 Host（debug 模式），确认两进程、单端口（Edge）、两日志目录独立。
   - 立刻调用 `GET /v1/ingest/queue/status`，记录响应作为证据：必须包含 `processing_mode` 且值为 `noop`。
   - 检查 Edge 启动日志：必须出现且仅出现一次明确行表明 `processing_mode=noop`（字面匹配：`MRV3 processing_mode=noop`）；且不得出现 OCR/provider/model preload 的初始化日志。
2. 访问 `/`、`/search`、`/timeline`，记录每个页面 HTTP 200 与首屏渲染截图。
3. 连续上传 50 条 capture（`POST /v1/ingest`，每条携带唯一 capture_id，使用 JPEG 文件），记录每条的 HTTP 响应与 ingest 状态流转。
   - 新建成功：期望 `201 Created` + `{"status":"queued"}`。
   - 若因网络抖动/重试导致幂等重放：允许出现 `200 OK` + `{"status":"already_exists"}`。
   - 状态流转口径（SSOT：`spec.md` §4.7）：`pending -> completed`（允许实现上经过 processing，但验收脚本不得依赖 processing 的可观测性）；P1-S1 不做 AX/OCR。
4. 人为制造断连（阻断 Host→Edge 连接 3 分钟），继续产生 capture（写入 Host buffer，不上传）。
5. 恢复连接，确认 Host 自动续传，断连期间 buffer 中的 capture 最终全部入队并收到 201（或 200 already_exists）。
   - 允许出现 `200 already_exists`（例如断连窗口边界存在请求是否到达的歧义；以幂等成功为准）。
6. 回放重复包（相同 capture_id 至少 10 条），核对去重结果：重复包返回 `200 OK + "status": "already_exists"`（可附 `request_id`，且不包含错误 `code` 字段），`frames` 表无重复行。
7. 调用 `GET /v1/ingest/queue/status`，核对 pending/processing/completed/failed 计数与实际 DB 状态一致；检查错误码与失败重试日志。
   - 若 `failed > 0`，必须给出失败原因分类；P1-S1 不接受 "AI/OCR/provider/model 初始化失败" 作为 failed 来源。
8. 校验 UI 健康态与错误态：
   - 判定口径（SSOT）：`spec.md` §4.8.1（强制 `#mr-health` + `data-state`）。
   - 证据要求：对 `/`、`/search`、`/timeline` 三个页面分别截图，截图中必须包含首屏可见的 `#mr-health` 组件。
   - 正常态：三页均应显示 `data-state="healthy"`，且文案包含 `服务健康/队列正常`。
   - 故障注入（UNREACHABLE，禁止刷新页面）：
     - 保持三页均已打开且不刷新页面，制造浏览器侧对 `GET /v1/health` 的请求失败/超时（例如停止 Edge 进程，或在 DevTools 中对 `/v1/health` 启用 request blocking）。
     - 在 15 秒内（三页）必须进入 `data-state="unreachable"`，且出现 `Edge 不可达`。
   - 恢复（禁止刷新页面）：
     - 恢复 `GET /v1/health` 可达（例如启动 Edge 进程，或解除 request blocking）。
     - 在 10 秒内（三页）必须自动恢复为 `data-state="healthy"`，且出现 `服务健康/队列正常`。
9. 图片格式契约检查：
   - 抽样调用 `GET /v1/frames/:frame_id`，响应头必须为 `Content-Type: image/jpeg`；
   - 抽样校验 `frames.snapshot_path` 指向 JPEG 文件（`.jpg`/`.jpeg`）；
   - 抽样校验 Host spool 新写入项为 JPEG（`.jpg`/`.jpeg`）+ `.json`；若存在 `.webp`，仅允许为历史遗留 drain 项；
   - 若启用兼容输入（PNG/WebP），验证入库后读取仍为 `image/jpeg`。
10. 命名空间回归检查（legacy 端点废弃回归）：
    - 对旧路径（`POST /api/upload`、`GET /api/search`、`GET /api/queue/status`、`GET /api/health`）发起请求，确认返回 **301 重定向**至对应 `/v1/*`
    - 响应头包含 `Location` 字段，值为对应的 `/v1/*` 路径（响应体可选包含 `new_location` 作为调试字段）
    - 检查服务器日志，确认每条请求记录了 `[DEPRECATED]` 标记（格式：`/api/{endpoint} -> /v1/{endpoint}`）
    - 验收脚本主流程仅允许引用 `/v1/*` 路径；旧命名空间废弃回归仅覆盖本节列出的 legacy 端点
    - 说明：对 `GET /api/search` 的 301 仅验证“命名空间迁移提示 + [DEPRECATED] 日志”，不要求本阶段 `/v1/search` 可用（可能为 404，属预期）。
