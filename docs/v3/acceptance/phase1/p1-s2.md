## 0. screenpipe 对照与对齐

- screenpipe 做法：事件驱动采集（app switch/click/typing/idle）并与上下文信息关联。
- **去抖对齐（已确认一致）**：
  - screenpipe 实际使用统一 `min_capture_interval_ms=200ms`（非文档描述的分层）
  - `idle_capture_interval_ms=30s`（与 MyRecall v3 一致）
  - P1 触发枚举：`idle/app_switch/manual/click`；`window_focus` 不纳入 P1
- **内容去重对齐（需实现）**：
  - screenpipe 使用 `content_hash`(AX树) + 30s 保底写入
  - MyRecall v3 当前：`capture_id` 幂等**规格已定义（待实现）**，`content_hash` 字段预留但**去重逻辑未实现**
  - P1 需实现：
    - Edge 根据 `capture_id` 做幂等写入（UNIQUE 约束 + INSERT OR IGNORE）
    - Edge 根据 `content_hash` 做内容去重（非 idle/manual）+ 30s 保底写入

## 1. 范围与目标

- 范围：Host 事件驱动 capture、idle fallback、AX 文本采集并上传（主链路图片格式为 JPEG）。
- 目标：验证触发覆盖、性能与数据完整性，同时让 timeline 可见上传/入队状态。
- 对应 Gate 条件：
  - 入队时延 Gate：压测窗口（5 分钟）`enqueue_latency_p95 <= 3s`（`eligible_events >= 200`）。
  - 每分钟 300 次事件压测下丢包率 < 0.3%。
  - 触发覆盖 Gate：`trigger_coverage = covered_trigger_types / 4 = 100%`（`idle/app_switch/manual/click` 四类均需命中；每类样本 >= 20）。
  - 去抖 Gate：同 monitor 连续 `app_switch/click` 入库间隔 < `min_capture_interval_ms`（默认 200ms）违规数 = 0。
  - 去重 Gate：重复内容压测窗口（5 分钟）满足 `dedup_skip_rate = dedup_skipped / dedup_eligible >= 95%`（`dedup_eligible >= 500`），且 `inter_write_gap_sec` 满足 `P99 <= 30s` 与 `max <= 45s`。
  - 背压 Gate：过载注入窗口（5 分钟）满足 `collapse_trigger_count >= 1`、`queue_saturation_ratio <= 10%` 且 `overflow_drop_count = 0`。
  - 新 capture 在 timeline 可见性通过率 >= 95%。

## 3. 验收步骤

1. 启动 Host/Edge，开启采集 debug 日志。
2. 按触发清单执行操作：
   - app switch（>= 50 次）
   - click（>= 100 次）
   - manual（>= 20 次）
   - idle fallback（>= 20 次）
3. 执行 300 events/min 压测 5 分钟，记录 Host CPU 峰值与平均值。
4. 对账触发日志与入队记录，计算 `enqueue_latency_p95` 与 Capture 丢失率。
5. 抽样核验 AX 文本采集字段（app/window/timestamp/trigger）完整性。
6. 执行高频“内容不变”压测（app switch/click，5 分钟），统计 `dedup_skip_rate` 并校验 `inter_write_gap_sec`（P99/max）。
7. 制造触发通道 lag（短时突发 > 处理能力，5 分钟），统计 `collapse_trigger_count`、`queue_saturation_ratio` 与 `overflow_drop_count`。
8. 执行 SQL 校验：
   - `enqueue_latency_p95`（目标 <= 3s）：
     - 以同一 capture 的 `event_ts` 与 `edge_enqueued_ts` 计算 `enqueue_latency_sec` 样本分布；
     - 判定：`eligible_events >= 200` 且 `P95(enqueue_latency_sec) <= 3`。
   - `trigger_coverage`（目标 = 100%）：
     - 统计窗口内 `idle/app_switch/manual/click` 四类触发类型命中情况；
     - 判定：四类均命中且每类样本 `>= 20`。
   - 去抖违规数（目标 0，同 monitor 以 `device_name` 分区）：
     ```sql
     WITH ordered AS (
       SELECT id, device_name, capture_trigger, timestamp,
              LAG(timestamp) OVER (
                PARTITION BY device_name
                ORDER BY timestamp
              ) AS prev_ts
       FROM frames
       WHERE capture_trigger IN ('app_switch', 'click')
     )
     SELECT COUNT(*) AS debounce_violations
     FROM ordered
     WHERE prev_ts IS NOT NULL
       AND ((julianday(timestamp) - julianday(prev_ts)) * 86400000.0) < 200;
     ```
   - `inter_write_gap_sec`（目标：P99 <= 30s 且 max <= 45s）：
     ```sql
     WITH samples AS (
       SELECT timestamp
       FROM frames
       WHERE capture_trigger IN ('app_switch', 'click')
         AND timestamp >= datetime('now', '-5 minutes')
     ),
     ordered AS (
       SELECT timestamp,
              LAG(timestamp) OVER (ORDER BY timestamp) AS prev_ts
       FROM samples
     )
     SELECT
       (SELECT COUNT(*) FROM samples) AS writes,
       COALESCE(MAX((julianday(timestamp) - julianday(prev_ts)) * 86400.0), 0) AS max_gap_sec
     FROM ordered
     WHERE prev_ts IS NOT NULL;
     ```
     判定：在“内容不变”压测窗口内，`writes >= 100` 且 `max_gap_sec <= 45`；同时输出 gap 样本分布并验证 `P99 <= 30`。
   - `dedup_skip_rate`（目标 >= 95%）：
     - 取压测窗口起止点的 `dedup_eligible_total`、`dedup_skipped_total` 计数器差值，按公式计算；
     - 判定：`dedup_eligible >= 500` 且 `dedup_skip_rate >= 95%`。
   - 背压指标（目标：`collapse_trigger_count >= 1`、`queue_saturation_ratio <= 10%`、`overflow_drop_count = 0`）：
     - 取过载注入窗口的队列深度时间序列与通道事件计数器；
     - 计算 `queue_saturation_ratio = (queue_depth >= 0.9 * queue_capacity 的采样数 / 总采样数) * 100%` 并给出判定。
9. 抽样校验本阶段新 capture：`frames.snapshot_path` 指向 JPEG 文件（`.jpg`/`.jpeg`），并通过 `GET /v1/frames/:frame_id` 返回 `Content-Type: image/jpeg`。
10. 抽样校验 Host spool 新写入项为 JPEG（`.jpg`/`.jpeg`）+ `.json`；若存在 `.webp`，仅允许为历史遗留 drain 项。
11. 打开 timeline，确认新 capture 的“上传中/已入队/失败重试”状态可见，且 dedup 场景下时间线无长时空洞。
