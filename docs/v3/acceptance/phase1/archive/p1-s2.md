## 1. 范围与目标

- 范围：Host 事件驱动 capture、idle fallback、AX 文本采集并上传（主链路图片格式为 JPEG）。
- 目标：验证触发覆盖、性能与数据完整性，同时让 timeline 可见上传/入队状态。
- 对应 Gate 条件：
  - 每分钟 300 次事件压测下丢包率 < 0.3%。
  - 触发覆盖 Gate：`trigger_coverage = (covered_trigger_types / 4) × 100% = 100%`（`idle/app_switch/manual/click` 四类均需命中；每类样本 >= 20）。
   - 去抖 Gate：同 monitor 连续 `app_switch/click` 入库间隔 < `min_capture_interval_ms`（默认 1000ms；有意偏离 screenpipe 200ms）违规数 = 0。
   - 去重实现验证（Soft KPI）：重复内容压测窗口（5 分钟）验证 `dedup_eligible` 与 `dedup_skipped` 计数器实现正确，`dedup_skip_rate` 仅记录数值用于生产可观测性（目标 >= 95%，但不作为 Hard Gate）
   - `inter_write_gap_sec`（Soft KPI + Hard Gate）：
  - Hard Gate: `max <= 45s`（样本 >= 100）
  - Soft KPI（non-blocking）：记录 P50/P90/P99 分布
  - 说明：P99 不设硬性阈值（dedup 保底写入场景下 P99 天然接近 30s，无区分度）
  - 背压 Gate：过载注入窗口（5 分钟）满足 `collapse_trigger_count >= 1`、`queue_saturation_ratio <= 10%` 且 `overflow_drop_count = 0`。
  - 新 capture 在 timeline 可见性通过率 >= 95%。

## 3. 验收步骤

1. 启动 Host/Edge，开启采集 debug 日志。
2. 按触发清单执行操作：
   - app switch（>= 50 次）
   - click（>= 100 次）
   - manual（>= 20 次）
   - idle fallback（>= 20 次）
3. 执行 300 events/min 压测 5 分钟，记录 Host CPU 峰值与平均值。
4. 对账触发日志与 Edge 持久化记录，计算 `capture_latency_p95` 与 Capture 丢失率。
5. 抽样核验 AX 文本采集字段（app/window/timestamp/trigger）完整性。
6. 执行高频“内容不变”压测（app switch/click，5 分钟），统计 `dedup_skip_rate` 并校验 `inter_write_gap_sec`（Soft KPI + Hard Gate: max <= 45s）。
7. 制造触发通道 lag（短时突发 > 处理能力，5 分钟），统计 `collapse_trigger_count`、`queue_saturation_ratio` 与 `overflow_drop_count`。
8. 执行 SQL 校验：
   - `capture_latency_p95`（观测记录，non-blocking）：
     - 以同一 capture 的 `event_ts` 与 `edge_db_persisted_ts` 计算 `capture_latency_ms` 样本分布；
     - 记录：P50/P90/P95/P99、样本量与时间窗（不作为本阶段 Gate Fail 条件）。
   - `trigger_coverage`（目标 = 100%）：
     - 统计窗口内 `idle/app_switch/manual/click` 四类触发类型命中情况；
     - 判定：四类均命中且每类样本 `>= 20`。
   - 去抖违规数（目标 0，同 monitor 以 `device_name` 分区）：
     ```sql
     WITH ordered AS (
       SELECT id, device_name, capture_trigger, timestamp,
              LAG(timestamp) OVER (
                PARTITION BY device_name
                ORDER BY timestamp
              ) AS prev_ts
       FROM frames
       WHERE capture_trigger IN ('app_switch', 'click')
     )
      SELECT COUNT(*) AS debounce_violations
      FROM ordered
      WHERE prev_ts IS NOT NULL
        AND ((julianday(timestamp) - julianday(prev_ts)) * 86400000.0) < 1000;
      ```
      > 注：阈值为 1000ms，对应 P1 默认 1 Hz 频率（有意偏离 screenpipe 200ms）
   - `inter_write_gap_sec`（Soft KPI + Hard Gate，历史草案）：
      - Hard Gate: `max <= 45s`（样本 >= 100）
      - Soft KPI（non-blocking）：记录 P50/P90/P99 分布
      - 说明：P99 不设硬性阈值（dedup 保底写入场景下 P99 天然接近 30s，无区分度）
      - ⚠️ 本段 SQL 为拆分前历史示例；现行 Gate 请使用 `p1-s2b.md` 中按 `device_name` 分桶的版本。
      ```sql
     WITH samples AS (
       SELECT timestamp
       FROM frames
       WHERE capture_trigger IN ('app_switch', 'click')
         AND timestamp >= datetime('now', '-5 minutes')
     ),
     ordered AS (
       SELECT timestamp,
              LAG(timestamp) OVER (ORDER BY timestamp) AS prev_ts
       FROM samples
     )
     SELECT
       (SELECT COUNT(*) FROM samples) AS writes,
       COALESCE(MAX((julianday(timestamp) - julianday(prev_ts)) * 86400.0), 0) AS max_gap_sec
     FROM ordered
     WHERE prev_ts IS NOT NULL;
     ```
     判定：在“内容不变”压测窗口内，`writes >= 100` 且 `max_gap_sec <= 45`；同时输出 gap 样本分布并验证 `P99 <= 30`。
    - `dedup_skip_rate`（Soft KPI，仅记录数值）：
      - 取压测窗口起止点的 `dedup_eligible_total`、`dedup_skipped_total` 计数器差值，按公式计算；
      - 记录实际数值用于生产可观测性（目标 >= 95%，但不作为判定条件）。
   - 背压指标（目标：`collapse_trigger_count >= 1`、`queue_saturation_ratio <= 10%`、`overflow_drop_count = 0`）：
     - 取过载注入窗口的队列深度时间序列与通道事件计数器；
     - 计算 `queue_saturation_ratio = (queue_depth >= 0.9 * queue_capacity 的采样数 / 总采样数) * 100%` 并给出判定。
9. 抽样校验本阶段新 capture：`frames.snapshot_path` 指向 JPEG 文件（`.jpg`/`.jpeg`），并通过 `GET /v1/frames/:frame_id` 返回 `Content-Type: image/jpeg`。
10. 抽样校验 Host spool 新写入项为 JPEG（`.jpg`/`.jpeg`）+ `.json`；若存在 `.webp`，仅允许为历史遗留 drain 项。
11. 打开 timeline，确认新 capture 的“上传中/已入队/失败重试”状态可见，且 dedup 场景下时间线无长时空洞。

## 7. P1-S2 Rollback Rule（Entry Contract）

### 7.1 触发条件（任一命中即触发回滚流程）

- 任一改动导致 P1-S1 防回归基线出现失败。
- 任一改动破坏已冻结契约（ingest/queue/health/UI health gate/timestamp 口径）。
- 任一改动导致关键 HTTP 契约返回码/字段语义偏离 [../../../spec.md](../../../spec.md) 与 openspec 记录。

### 7.2 回滚流程（必须按顺序执行）

1. 停止继续叠加新改动，冻结当前提交窗口。
2. 定位引入回归的最小变更集（优先按最近变更文件与测试失败映射）。
3. 撤销或修复该最小变更集，直到 P1-S1 防回归基线恢复全绿。
4. 重新执行组合回归并保存证据文件到 `docs/v3/acceptance/phase1/evidence/`。

### 7.3 解除阻塞条件

- 仅当 P1-S1 防回归基线恢复为 `0 failed`，且本次 P1-S2 目标测试同样 `0 failed`，才允许继续下一个开发任务。
