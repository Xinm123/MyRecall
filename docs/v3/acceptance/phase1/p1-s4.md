## 0. screenpipe 对照与对齐（020A）

- screenpipe 做法：`GET /search`（底层纯 FTS5，无 embedding 参与）+ `GET /search/keyword`（独立快速路径）。
- 本阶段对齐（020A）：线上检索仅 FTS5+过滤，不引入 hybrid；`/v1/search/keyword` 合并到 `/v1/search`（P1 无 embedding，拆分无意义）；对齐 API query params 与 response schema。
- 口径约束（Scheme C SSOT）：检索按 `content_type` 三路径分发（`search_ocr()` / `search_accessibility()` / `search_all()`）；不得用“`completed` 帧与 `ocr_text` 行数相等”替代分表一致性校验。

## 1. 范围与目标

- 范围：`/v1/search`（含原 keyword 路径，已合并）、返回字段契约与 UI 过滤交互。（020A：`/v1/search/keyword` 在 P1 不作为独立端点）
- 目标：确保检索准确、可过滤、可回溯，并验证 UI 与 API 参数映射一致。
- 对应 Gate 条件：
  - 精确词查询不低于对齐基线。
  - 观测 KPI：Search P95 记录实际分布（标准时间窗 <= 24h，P1 阶段暂不设硬性阈值，参考 screenpipe 5s 问题阈值）。
  - `/v1/search` 过滤参数契约完成率 = 100%。
  - Search SQL 三路径分发一致性 = 100%（`search_ocr`: `frames INNER JOIN ocr_text`；`search_accessibility`: `accessibility` + `accessibility_fts` + `LEFT JOIN frames`；`search_all`: 并行合并）。
  - FTS 清空一致性 = 100%（字段由非空更新为 `NULL/''` 后，旧 token 不可再检索命中）。
  - 分表完整性一致性 = 100%（按 `text_source` 对账，`missing_ocr=0` 且 `missing_ax=0`）。
  - 分表互斥性一致性 = 100%（`dual_written=0`，同一帧不同时写入 `ocr_text` 与 `accessibility(frame_id)`）。
  - completed 帧覆盖一致性 = 100%（`uncovered=0`，任一 completed 帧至少被 OCR/AX 其中一路覆盖）。
  - `/v1/search/keyword` 不作为独立端点暴露（请求返回 `404 Not Found`）。
- 旧命名空间废弃回归仅覆盖 legacy 端点：`POST /api/upload`、`GET /api/search`、`GET /api/queue/status`、`GET /api/health`。P1-S4 起必须返回 `410 Gone`（从 P1-S1 的阶段性重定向策略：POST=308、GET=301，切换为完全废弃）。
  - OCR 检索结果引用字段完整率 = 100%（`frame_id`/`timestamp`，Hard Gate）。
  - UI 检索结果引用字段完整率 = 100%（`frame_id`/`timestamp`，Hard Gate，P1 阶段 accessibility.frame_id 非 NULL）。
  - 观测 KPI（non-blocking）：OCR 检索结果 `capture_id` 覆盖率目标 >= 99%（未达标需提交整改动作）。
  - Search UI 过滤项契约映射完成率 = 100%。
  - 检索结果点击回溯成功率 >= 95%。

## 3. 验收步骤

1. 导入标准检索数据集，记录基线版本。
2. 运行精确词查询集，对比基线召回与排序。
3. 对 `/v1/search` 执行过滤组合测试（单参、多参、边界时间窗）。
4. 通过 SQL 日志或 `EXPLAIN QUERY PLAN` 抽样核对三路径：
   - `content_type=ocr`：主路径 `frames INNER JOIN ocr_text`；`frames_fts` 仅在 app/window/browser/focused 非空时追加；`ocr_text_fts` 仅在 `q` 非空时追加。
   - `content_type=accessibility`：主路径 `accessibility` + `accessibility_fts`（按条件）+ `LEFT JOIN frames`。
   - `content_type=all`：并行执行 OCR/AX 两路径并按 timestamp 合并分页。
5. 执行 Scheme C 一致性核对 SQL（四项结果都必须为 0）：
   - `missing_ocr`：
     ```sql
     SELECT COUNT(*) AS missing_ocr
     FROM frames f
     LEFT JOIN ocr_text o ON o.frame_id = f.id
     WHERE f.status='completed' AND f.text_source='ocr' AND o.frame_id IS NULL;
     ```
   - `missing_ax`：
     ```sql
     SELECT COUNT(*) AS missing_ax
     FROM frames f
     LEFT JOIN accessibility a ON a.frame_id = f.id
     WHERE f.status='completed' AND f.text_source='accessibility' AND a.id IS NULL;
     ```
   - `dual_written`：
     ```sql
     SELECT COUNT(*) AS dual_written
     FROM frames f
     JOIN ocr_text o ON o.frame_id = f.id
     JOIN accessibility a ON a.frame_id = f.id;
     ```
   - `uncovered`：
     ```sql
     SELECT COUNT(*) AS uncovered
     FROM frames f
     LEFT JOIN ocr_text o ON o.frame_id = f.id
     LEFT JOIN accessibility a ON a.frame_id = f.id
     WHERE f.status='completed' AND o.frame_id IS NULL AND a.id IS NULL;
     ```
6. 验证 `/v1/search` 的关键词路径性能（`/v1/search/keyword` 已合并，P1 不作为独立端点）。
7. 对 `GET /v1/search/keyword` 发起请求，验证返回 `404 Not Found`（避免独立端点回归）。
8. 旧命名空间废弃检查（P1-S4 切换项，legacy 端点废弃回归）：
   - 对旧路径（`POST /api/upload`、`GET /api/search`、`GET /api/queue/status`、`GET /api/health`）发起请求；
   - 确认统一返回 `410 Gone`（不再返回 `301`）；
   - 记录响应样本与日志证据，作为 S4 命名空间 Hard Gate 依据。
9. 按结果类型抽样校验引用字段完整性：
   - `type=OCR`：`frame_id` 与 `timestamp` 同时非空；
   - `type=UI`：`frame_id` 与 `timestamp` 同时非空（P1 阶段正常 paired_capture 路径下 accessibility.frame_id 应为非 NULL）；
   - 统计 `type=OCR` 的 `capture_id` 覆盖率（non-blocking 观测项）。
10. 在 Search 页面逐项操作过滤器，抓包核对 UI 参数与 API 请求 1:1 对应。
11. 从检索结果点击回溯到 frame/timeline，统计成功率。
12. 执行 FTS clear-safe 回归（`frames_fts` + `ocr_text_fts`）：
   - 构造样本（包含 `app_name/window_name/browser_url/accessibility_text/text` 的可检索 token）。
   - 对样本行执行 UPDATE，将上述字段改为 `NULL` 或 `''`。
   - 立即执行对应 MATCH 查询，验证旧 token 命中数 = 0。
   - 再次写入新 token 并查询，验证新 token 可命中（防止误删后不可恢复）。
