## 0. screenpipe 对照与对齐（020A）

- screenpipe 做法：`GET /search` 基于纯 FTS5，无 embedding 参与；`GET /search/keyword` 为独立快速路径。
- 本阶段对齐（020A）：线上检索仅 FTS5+过滤，不引入 hybrid；`/v1/search/keyword` 合并到 `/v1/search`（P1 无 embedding，拆分无意义）。
- v3 主线约束：检索按 OCR-only 单路径收口，不验证 `search_accessibility()` / `search_all()` / `content_type=accessibility/all`。

## 1. 范围与目标

- 范围：`/v1/search`（含原 keyword 路径，已合并）、返回字段契约与 UI 过滤交互。
- 目标：确保检索准确、可过滤、可回溯，并验证 UI 与 API 参数映射一致。
- 对应 Gate 条件：
  - 精确词查询不低于对齐基线。
  - 观测 KPI：Search P95 记录实际分布（标准时间窗 <= 24h，P1 阶段暂不设硬性阈值）。
  - `/v1/search` 过滤参数契约完成率 = 100%。
  - OCR 搜索 SQL/返回结构一致性 = 100%。
  - FTS 清空一致性 = 100%。
  - OCR-only 可检索完整性一致性 = 100%。
  - `/v1/search/keyword` 不作为独立端点暴露（请求返回 `404 Not Found`）。
  - Search 引用字段完整率 = 100%（`frame_id`/`timestamp`，Hard Gate）。
  - 观测 KPI（non-blocking）：Search `capture_id` 覆盖率目标 >= 99%。
  - Search UI 过滤项契约映射完成率 = 100%。
  - 检索结果点击回溯成功率 >= 95%。

## 3. 验收步骤

1. 导入标准检索数据集，记录基线版本。
2. 运行精确词查询集，对比基线召回与排序。
3. 对 `/v1/search` 执行过滤组合测试（单参、多参、边界时间窗）。
4. 通过 SQL 日志或 `EXPLAIN QUERY PLAN` 抽样核对单路径 OCR 查询：
   - 主路径 `frames INNER JOIN ocr_text`
   - `frames_fts` 仅在 app/window/focused 非空时追加；`browser_url` 在 P1 不作为 active 过滤能力 Gate
   - `ocr_text_fts` 仅在 `q` 非空时追加
5. 执行 OCR-only 一致性核对 SQL：
   - `missing_ocr`：
     ```sql
     SELECT COUNT(*) AS missing_ocr
     FROM frames f
     LEFT JOIN ocr_text o ON o.frame_id = f.id
     WHERE f.status='completed' AND f.text_source='ocr' AND o.frame_id IS NULL;
     ```
   - `unexpected_accessibility_rows`（v4 seam 保护检查）：
     ```sql
     SELECT COUNT(*) AS unexpected_accessibility_rows
     FROM accessibility;
     ```
     > **检查目的**：验证 v4 reserved seam 未被 v3 代码意外写入。OCR-only 主线下该值应为 0。
   - `uncovered`：
     ```sql
     SELECT COUNT(*) AS uncovered
     FROM frames f
     LEFT JOIN ocr_text o ON o.frame_id = f.id
     WHERE f.status='completed' AND f.text_source='ocr' AND o.frame_id IS NULL;
     ```
   判定规则：`missing_ocr=0 AND unexpected_accessibility_rows=0 AND uncovered=0`。
6. 验证 `/v1/search` 的关键词路径性能（`/v1/search/keyword` 已合并，P1 不作为独立端点）。
7. 对 `GET /v1/search/keyword` 发起请求，验证返回 `404 Not Found`（避免独立端点回归）。
8. 旧命名空间废弃检查：
   - 对旧路径（`POST /api/upload`、`GET /api/search`、`GET /api/queue/status`、`GET /api/health`）发起请求；
   - 确认统一返回 `410 Gone`；
   - 记录响应样本与日志证据。
9. 抽样校验引用字段完整性：
   - 搜索结果中的 `frame_id` 与 `timestamp` 同时非空；
   - `capture_id` 覆盖率作为 non-blocking 观测项统计。
10. 在 Search 页面逐项操作过滤器，抓包核对 UI 参数与 API 请求 1:1 对应。
11. 从检索结果点击回溯到 frame/timeline，统计成功率。
12. 执行 FTS clear-safe 回归（`frames_fts` + `ocr_text_fts`）：
   - 构造样本（包含 `app_name/window_name/focused/text` 的可检索 token；`browser_url` 仅验证 clear-safe reserved 列不产生脏索引）。
   - 对样本行执行 UPDATE，将上述字段改为 `NULL` 或 `''`。
   - 立即执行对应 MATCH 查询，验证旧 token 命中数 = 0。
   - 再次写入新 token 并查询，验证新 token 可命中。
