## 0. screenpipe 对照与对齐

- screenpipe 做法：强调从数据采集到用户交互的闭环可用性与持续回归。
- 本阶段对齐：以关键用户路径做 E2E 验收与故障注入回归；不追求同拓扑实现。
- 注：P1 OCR-only 范围内，不含结构化 citation 后处理；引用覆盖率评估为 Soft KPI 观测（non-blocking）。

## 1. 范围与目标

- 范围：P1 全链路（capture -> OCR processing -> search -> chat -> citation 回溯）端到端验收。
- 目标：确认 P1 OCR-only 功能冻结基线可用、可恢复、可追溯，并为 P2/P3 提供稳定起点。
- 对应 Gate 条件：
  - OCR主路径 TTS P95 <= 15s（Soft KPI，观测记录，不触发 Gate Fail）。
  - `capture_latency_p95` 观测记录完整（P50/P90/P95/P99 + 样本量 + 时间窗），不参与 Gate Pass/Fail 判定。
  - S1~S6 回归全通过。
  - P1 功能清单完成率 = 100%。
  - P1 验收记录完整率 = 100%。
  - UI 关键路径脚本通过率 = 100%。
  - 观测 KPI（non-blocking）：Chat 引用覆盖率目标 >= 92%，Stretch >= 95%。

## 3. 验收步骤

1. 逐项确认 S1~S6 前置 Gate 结果均为 Pass。
2. 执行关键路径脚本：`timeline -> search -> chat -> citation -> frame`。
3. 统计 OCR主路径 TTS（capture 到可检索）P95。
4. 汇总 P1-S2a 留存的 `capture_latency_p95` 观测数据（P50/P90/P95/P99 + 样本量 + 时间窗）并归档证据。
5. 运行 chat 问题集，统计基于 deep link 的引用覆盖率并抽检真实性（`myrecall://frame/{id}?timestamp=...`；ID/时间戳不得伪造）；同时验证引用点击后统一落到 `/timeline`（`frame` 链接通过 `/v1/frames/:frame_id/metadata` 解析 timestamp 定位）。引用覆盖率为 Soft KPI 观测（non-blocking）；若有后续低覆盖率整改，方向为 Skill prompt 优化或 P2 增强。
6. 执行故障注入场景，验证链路可恢复与错误可见。
7. 汇总功能清单、契约清单、异常场景清单完成度。
8. 归档证据：日志、截图、录屏、指标导出。
