## 0. screenpipe 对照与对齐

- screenpipe 做法：通过 Pi `--provider`/`--model` 启动参数 + `models.json` 配置进行 provider 切换；Pi stdout 以 JSON Lines 事件流式输出；无 auto-fallback chain。
- 本阶段对齐：对齐 Pi provider 路由机制和流式事件协议（DA-5：P1 不做自动 fallback）；通过 PiManager 重启 Pi 进程实现 provider 切换。

## 1. 范围与目标

- 范围：模型路由（provider/model 配置切换）、Pi 事件流式输出（SSE 透传）、Provider timeout 处理（180s 请求 watchdog）。
- 目标：在故障注入下仍可稳定返回，并让 UI 感知路由状态与 timeout 过程。P1 不实现 auto-fallback（DA-5，对齐 screenpipe）。
- 对应 Gate 条件：
  - Chat 系统可用率 >= 98%（仅 Pi crash、Manager 协议错误、Edge 内部 500 计入失败；provider 5xx/429、180s timeout 不计入失败）。
  - Provider 切换可重复通过。
  - Provider timeout 处理可重复通过（180s 请求 watchdog → timeout error；不强制 abort）。
  - 观测 KPI（non-blocking）：Chat 首 token P95 <= 3.5s。
  - 路由切换场景覆盖率 = 100%（provider 切换 + timeout 错误，不含 auto-fallback）。
  - 流式输出协议一致性用例通过率 = 100%。
  - 路由与 timeout 状态可见场景覆盖率 = 100%。

## 3. 验收步骤

1. 分别配置不同 provider（如 OpenAI、Ollama），通过 UI 配置页面切换并验证 PiManager 重启 Pi 进程。
2. 在正常网络下运行问题集，统计 Chat 请求成功率（并可选输出首 token P95观测值）。
3. 注入 provider 超时（>180s 无成功响应），验证返回 timeout error event（不强制 abort）。
4. 注入 Pi 进程 crash（SIGKILL），验证 watchdog 自动重启并返回 interrupt frame。
5. 验证流式输出协议（分层口径）：
   - 顶层 `data.type` 顺序与终止规则：`message_start → message_update* → message_end`，`turn_start` 包含 `agent_start/end`，`response` 为终止事件；
   - `message_update.assistantMessageEvent.type` 子事件解析：至少覆盖 `text_delta`、`thinking_start`、`thinking_delta`、`thinking_end`、`content_block_delta`；
   - 未知扩展顶层事件的前向兼容行为：不崩溃、可记录、可降级展示。
6. 在 UI 中核验：当前 provider/model badge、timeout/error notification、Pi 健康状态指示器。
7. 重复故障注入场景至少 3 轮，确认结果可复现。

## 6. 风险与后续动作

- 风险：provider 不稳定导致 Chat 完成率低于目标，但不触发系统可用率 Gate Fail。
- 缓解：完成率作为 Soft KPI 持续观测，低于 95% 时需给出整改动作（如优化 prompt 减少 token、调整 provider 配置）。
- 后续动作：将系统错误场景（Pi crash、协议错误）加入 P1-S7 E2E 固定回归。
