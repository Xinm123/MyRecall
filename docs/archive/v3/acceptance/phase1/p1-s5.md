## 0. screenpipe 对照与对齐

- screenpipe 做法：Pi agent 通过 SKILL.md 定义的工具（`screenpipe-search`）获取上下文，并输出可点击 deep link 作为引用来源（默认 `screenpipe://frame/{frame_id}`，缺少 `frame_id` 时回退 `screenpipe://timeline?timestamp=...`；prompt-driven citation，无结构化解析，禁止伪造 ID/时间戳）。
- 本阶段对齐：Pi Sidecar + Python Manager 架构（DA-7=A），通过 `myrecall-search` SKILL.md 实现 tool-driven retrieval；引用采用提示词驱动 deep link（DA-8=A），默认输出 `myrecall://`。

## 1. 范围与目标

- 范围：Pi Sidecar 基础能力（进程管理、RPC 通信、SKILL.md 工具调用）、grounding 与引用链路。
- 目标：确保"有依据回答"成为默认路径，引用在 UI 可点击回溯。
- 对应 Gate 条件：
  - **Pi 可用性前置检查**（Hard Gate）：bun --version 可执行，pi --version 可执行
  - Chat 工具能力清单完成率 = 100%。
  - Chat 引用点击回溯成功率 >= 95%。
  - 观测 KPI（non-blocking）：Chat 引用覆盖率目标 >= 85%（未达标需提交整改动作）。

## 3. 验收步骤

1. 运行 grounding 问题集，记录每轮回答中的 deep link 原文与解析结果。
2. 统计引用覆盖率，抽检 deep link 中的 `frame_id`/`timestamp` 是否直接来自检索结果且可回溯成功。
3. 对不可回答问题集验证：
   - 无依据时应拒答/标记低置信；
   - 不得生成伪造引用。
4. 检查 SKILL.md 工具调用覆盖：`myrecall-search` Skill 完整能力被触发验证（含搜索、时间范围策略、帧详情获取）。
5. 在 UI 中点击每条 `myrecall://` 引用，验证统一落到 `/timeline` 回溯（`myrecall://frame/{id}` 需通过 `/v1/frames/:frame_id/metadata`（timestamp resolver，最小稳定契约）解析 timestamp 后定位；无 frame_id 时走 `myrecall://timeline?timestamp=...`）。
6. 注入检索失败场景，验证错误可见性与可恢复行为。
