# Chat MVP Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the reduced MyRecall v3 chat MVP defined in `docs/v3/chat/mvp.md`, including schema changes, frame-backed accessibility search, Host-facing summary/search/context APIs, and OCR completion semantics.

**Architecture:** The implementation keeps OCR on the existing async server worker path, adds paired accessibility capture data to the server data plane, and exposes a screenpipe-style but reduced tool surface. `frames` becomes the frame context truth source, `accessibility` becomes a frame-backed search plane, and `elements` becomes an internal structured accessibility projection for summary generation.

**Tech Stack:** Python 3.11, Flask, SQLite/FTS5, pytest

---

## Implementation Phases

The work should be staged so accessibility acquisition is introduced incrementally, measured early, and only promoted into canonical persistence after the capture-side behavior is understood.

### Phase 0: Lock Contracts And Reason Vocabulary

**Goal:** Freeze the accessibility-side contracts before touching runtime code.

**Includes:**

- `docs/v3/chat/mvp.md` is the active spec for:
  - focused-window-only acquisition
  - focused-monitor-only AX eligibility in multi-monitor mode
  - terminal-class `app_prefers_ocr`
  - `TreeSnapshot` shape
  - flat depth-first `tree_json`
  - accessibility-canonical metadata payload
  - ingest split (`accessibility-complete` vs `OCR-pending`)
- keep the reason vocabulary stable:
  - `non_focused_monitor`
  - `app_prefers_ocr`
  - `no_focused_window`
  - `empty_text`
  - `adopted_accessibility`

**Why first:** Later code and tests will depend on these names and boundaries.

### Phase 1: Reset Schema And Persistence Boundaries

**Goal:** Make the database ready for accessibility-backed frames before any capture-side implementation lands.

**Includes:**

- `frames.accessibility_text` + `frames.ocr_text`
- `frames.accessibility_tree_json`
- metadata-only `frames_fts`
- frame-backed `accessibility + accessibility_fts`
- internal `elements`

**Exit criteria:**

- schema tests pass
- `FramesStore` has room for `complete_accessibility_frame(...)`
- no legacy unified `frames.text` column remains

### Phase 2: Add Accessibility Types, Policy, And Debug Scaffolding On The Client

**Goal:** Introduce the accessibility subsystem without changing capture semantics yet.

**Recommended files:**

- Create: `openrecall/client/accessibility/types.py`
- Create: `openrecall/client/accessibility/policy.py`
- Create: `openrecall/client/accessibility/debug.py`

**Includes:**

- `TreeWalkerConfig`
- `TreeSnapshot`
- `AccessibilityTreeNode`
- `AccessibilityDecision`
- `app_prefers_ocr(...)`
- AX eligibility helpers
- structured logging helpers
- debug JSON dump helpers

**Exit criteria:**

- pure policy tests pass
- debug output format is stable
- no recorder hot-path changes yet beyond imports and wiring stubs

### Phase 3: Integrate Accessibility Decision Stage Into `recorder.py` Without Adoption

**Goal:** Insert the accessibility stage into the capture hot path, but initially use it for decision logging only.

**Includes:**

- add the capture-order stage:
  - screenshot
  - active context
  - AX decision
  - metadata build
  - enqueue
- call `collect_for_capture()` directly in the hot path (no wrapper function)
- do not yet persist canonical accessibility payload to the server
- log and dump decisions for real captures

> **实际状态:** Phase 3 完成形态是内联调用 `collect_for_capture()`，不引入 `_collect_accessibility_decision()` 包装函数。metadata merge (`_merge_accessibility_metadata`) 延迟到 Phase 5 引入。

**Why this phase exists:** It lets you validate monitor routing, terminal gating, and timing costs before AX adoption mutates server-side behavior.

**Exit criteria:**

- logs show correct `non_focused_monitor` behavior in multi-monitor mode
- terminal apps are rejected with `app_prefers_ocr`
- capture still enqueues normally when AX collection fails

### Phase 4: Implement Minimal macOS Walker For Focused-Window Text Snapshot

**Goal:** Get a bounded Python macOS walker returning `TreeSnapshot` with `text_content` and light node lists.

**Recommended files:**

- Create: `openrecall/client/accessibility/macos.py`
- Create: `openrecall/client/accessibility/service.py`

**Includes:**

- focused-window lookup
- bounded tree walk
- skip-role filtering
- text-bearing role extraction
- `value -> title -> description` priority
- best-effort `bounds`
- best-effort `browser_url` for app names containing `safari` or `chrome` using `AXDocument` only
- `content_hash` and `simhash`

**Important constraint:**

- the first walker must return a flat depth-first node list, not a full raw tree

**Exit criteria:**

- adopted snapshots exist for at least one known-good app
- empty or failed walks degrade cleanly
- hot-path timing metrics exist for `ax_walk_ms`

### Phase 5: Promote Accessibility Snapshots Into Canonical Ingest Path

**Goal:** Start uploading accessibility-canonical metadata and let ingest complete frames synchronously.

**Includes:**

- `_merge_accessibility_metadata(...)` in `recorder.py`
- accessibility-canonical metadata payload upload
- `/v1/ingest` split:
  - accessibility-complete path
  - OCR-pending path
- one-transaction `complete_accessibility_frame(...)`

**Exit criteria:**

- accessibility-canonical frames arrive server-side already completed
- malformed AX payload degrades to OCR-pending instead of failing ingest
- duplicate ingest stays idempotent

### Phase 6: Expand Snapshot Persistence Into `elements` And Frame Context Inputs

**Goal:** Turn uploaded `tree_json` into the durable structures needed for chat grounding.

**Includes:**

- parse `tree_json`
- reconstruct `elements.parent_id` and `sort_order` from depth-first ordering
- persist `frames.accessibility_tree_json`
- use snapshot text as `frames.text`  ← **注：** 实际为 `frames.accessibility_text`（规范字段名）

**Exit criteria:**

- `elements` rows exist for accessibility-complete frames
- `frames.accessibility_tree_json` supports context assembly
- `activity-summary.recent_texts` has a trustworthy source

### Phase 7: Upgrade /v1/search To Content-Type Aware

**Goal:** Upgrade the search endpoint to support OCR, accessibility, and merged search modes.

**Depends on:**

- Phase 5 (accessibility-canonical frames with text_source)

**Includes:**

- `GET /v1/search` with `content_type = ocr | accessibility | all`
- Split SearchEngine into content-type-specific paths
- Typed union response entries
- `all` search ordering and non-duplication semantics

**Why this phase:** Search is the primary chat tool and should be delivered first. The current OCR-only search is insufficient for accessibility-canonical frames.

### Phase 8: Add /v1/activity-summary

**Goal:** Provide activity overview for chat agents.

**Depends on:**

- Phase 6 (query helpers for activity summary)
- Phase 7 (search endpoint pattern established)

**Includes:**

- `GET /v1/activity-summary`
- Apps aggregation from completed frames
- Recent texts from accessibility elements
- Audio summary as empty shell

**Why this phase:** Activity summary gives chat agents a broad overview before targeted search. It depends on elements table population from Phase 6.

### Phase 9: Add /v1/frames/{id}/context

**Goal:** Provide detailed frame context for chat grounding.

**Depends on:**

- Phase 5 (accessibility_tree_json persistence)
- Phase 6 (get_frame_context query helper)

**Includes:**

- `GET /v1/frames/{id}/context`
- Node parsing from accessibility_tree_json
- URL extraction from link-like nodes and text
- OCR fallback when accessibility unavailable

**Why this phase:** Frame context is the main evidence layer for chat answers. It should come after search and summary are available.

### Phase 10: Validate Real-World Accessibility Coverage Before Wider Rollout

**Goal:** Confirm the Python walker is good enough before considering concurrency or native reimplementation.

**Includes:**

- measure `ax_walk_ms`, `total_ms`, and timeout/truncation rates
- validate Safari, Chrome, and at least one non-browser GUI app
- validate multi-monitor focused/non-focused behavior
- validate terminal OCR preference behavior

**Decision gate:** Only after this phase should you decide whether the Python walker is sufficient or needs a lower-level implementation.
