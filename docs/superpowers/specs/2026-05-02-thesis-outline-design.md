# 毕设报告大纲设计文档

**课题**：基于端侧大模型的个人回忆助手系统实现  
**日期**：2026-05-02  
**版本**：v1.6

---

## 1. 设计定位

### 1.1 课题类型

毕业设计/论文兼顾型，以**系统实现**为载体，以**核心创新**为主线。既有完整的工程实现（采集-存储-检索-推理-交互全链路），又有明确的技术创新点（渐进式记忆推理）。

### 1.2 核心创新主线

**核心矛盾**：端侧大语言模型的上下文窗口有限 vs 个人数字记忆的海量持续积累。

**核心创新**：Skills 驱动的渐进式记忆推理机制 —— 通过四级递进的信息披露策略（活动概览 → 混合检索 → 帧级上下文 → 原始视觉帧），在有限的上下文窗口内高效利用海量记忆数据。

### 1.3 差异化定位（与现有产品对比）

| 维度 | Windows Recall / Rewind AI | MyRecall |
|------|---------------------------|----------|
| 架构形态 | 封闭产品 | **个人回忆助手 + 面向Agent的记忆服务** |
| Agent 集成 | 无外部集成能力 | **Skills 规范 + API 对外服务** |
| 记忆推理 | 一次性注入或简单检索 | **渐进式信息披露** |
| 隐私策略 | 部分依赖云端 | 全本地闭环 |

### 第2章 相关工作

**目标**：梳理领域研究现状，分析现有方案局限，建立本文切入点。

- **2.1 个人数字记忆系统**
  - Windows Recall：系统集成方案，全天候高频截图导致存储膨胀（几十至上百GB），隐私安全争议（数据可被恶意软件读取），功能封闭（无外部接口）
  - Rewind AI：商业产品，全时捕获与自然语言检索，闭源且记忆数据锁定在产品内部，无外部 Agent 集成能力
  - Screenpipe：开源基础设施，提供屏幕录制、多模态数据采集与基础检索的底层框架。主要聚焦于数据采集与存储，缺乏对采集内容的深度语义理解（如VLM语义摘要、多模态向量嵌入）
  - **分析重点**：Recall/Rewind 为封闭产品，记忆数据不对外暴露，无外部 Agent 集成能力；Screenpipe 为底层框架，缺乏语义理解与推理能力

- **2.2 端侧大语言模型部署与上下文约束**
  - 轻量化与量化技术
  - 端侧推理性能优化
  - 上下文窗口的硬性约束及其对长文本处理的影响

- **2.3 长期记忆增强的智能体架构**
  - MemGPT：虚拟上下文管理
  - EgoMem：全双工全模态终身记忆 Agent
  - HiMeS：海马体启发的个性化记忆系统
  - **分析重点**：现有研究聚焦于 Agent 自身的记忆管理，而非为外部 Agent 提供记忆服务
  - **待补充**：Mem0 、MemOS等记忆系统（后续深入讨论）

- **2.4 检索增强生成与混合检索**
  - RAG 基本框架
  - 全文检索引擎（FTS5、Elasticsearch 等）
  - 向量检索与近似最近邻搜索（ANN）
  - 混合检索与结果融合策略（RRF、加权融合等）

- **2.5 现有方案的局限性分析**
  - 局限1：现有产品/系统均为封闭架构，记忆数据不对外暴露，无法被外部 Agent 调用
  - 局限2：现有系统的检索-推理链路多为一次性注入，缺乏渐进式压缩策略
  - 局限3：现有端侧系统缺乏对海量个人记忆的高效组织与检索机制
  - **本文切入点**：设计 Skills 驱动的统一记忆架构，既实现个人回忆助手，又向外部 Agent 开放记忆服务，通过渐进式信息披露解决上下文窗口约束

**篇幅预估**：6-8页（约3000-4000字）

### 第3章 系统总体架构

**目标**：阐述系统整体设计，建立 Host-Edge 分离架构，明确 MyRecall 既作为个人回忆助手系统，又作为面向Agent的记忆服务的统一架构定位。

- **3.1 设计原则**
  - 端侧优先：全本地闭环，数据不上云
  - 隐私保护：用户完全掌控自己的数据
  - 采集-计算解耦：Host 负责采集，Edge 负责处理与推理
  - 开放可扩展：通过 API + Skills 为外部 Agent 提供记忆服务

- **3.2 Host-Edge 分离架构**
  - Host（采集端）：轻量级感知进程，屏幕捕获、文本提取、数据上传
  - Edge（计算端）：存储索引、语义检索、模型推理
  - 通信机制：REST API + 磁盘队列（Spool）解耦，幂等性保障

- **3.3 MyRecall 的架构定位：个人回忆助手与面向Agent的记忆服务**
  - 与 Windows Recall / Rewind 的架构定位对比
    - Recall / Rewind：封闭产品，记忆数据锁定在应用内部，无外部集成能力
    - MyRecall：**个人回忆助手** + **面向Agent的记忆服务**，以 Skills 规范为统一接口
  - 架构设计的核心思想：摒弃传统"底层服务+上层应用"的分层思路
  - 统一接口概述：所有记忆访问（包括内置助手的对话推理与外部 Agent 的调用）均通过同一套 Skills 规范与 REST API 进行
  - **定位声明**：MyRecall 以 Skills 规范为统一接口，**既直接为用户提供个人回忆助手的交互体验，又可面向外部 Agent 框架开放个人记忆服务能力**。两种形态在架构上完全平级，不存在分层或特权关系

- **3.4 系统数据流与模块职责边界**
  - 从屏幕捕获到自然语言回答的完整链路
  - 各模块的职责边界与接口定义
  - 数据流图（可复用中期报告中的图）

**篇幅预估**：5-6页（约2500-3000字）

### 第4章 事件驱动的屏幕信息采集

**目标**：回答"如何在采集端控制数据流入量，降低冗余？"

- **4.1 混合触发机制**
  - 事件触发：应用切换、窗口焦点变化、键鼠活跃度等
  - 定时触发：idle 状态回退，确保长周期内记忆连续性
  - 两种机制的互补设计

- **4.2 三级防抖架构**
  - click debounce → trigger debounce → capture debounce
  - 基于 AtomicInt ctypes 实现
  - 独立配置监听线程，SQLite > TOML 优先级，配置热重载
  - 三层防抖间隔均可动态更新而无需重启进程

- **4.3 多级去重策略**
  - 内容哈希精确去重：对 AX 文本内容计算 SHA-256，相同哈希直接丢弃
  - 感知哈希模糊去重：DCT 变换构建 pHash，汉明距离度量视觉相似度
    - 公式：$d(h_1, h_2) = \text{bit\_count}(h_1 \oplus h_2)$
  - 安全阀机制：超过最大跳过时长强制捕获，避免记忆断层

- **4.4 采集端性能与资源控制**
  - CPU / 内存开销控制
  - 后台进程的稳定性保障

**篇幅预估**：5-6页（约2500-3000字）

### 第5章 多源内容理解与语义提取

**目标**：回答"如何从屏幕截图中提取并理解高质量语义信息？"

- **5.1 文本提取层**
  - **5.1.1 无障碍接口优先策略（AX-first）**
    - macOS AX API 调用与有界深度优先遍历
    - 跳过装饰元素，提取有文本信息的节点内容
    - 场景适用性判断：仅对聚焦窗口采集，终端应用标记为 OCR 偏好
    - AX 文本通过检查后直接上传，跳过服务端 OCR
  - **5.1.2 OCR 视觉识别回退**
    - 适用场景：复杂 UI 界面、终端应用、AX 遍历失败
    - OCR 处理器异步执行
    - 待处理状态的帧入库后由服务端 OCR Worker 处理
  - **5.1.3 双源文本提取质量评估**
    - 来源标注：AX（结构化程度高）、OCR（视觉识别）
    - 质量评估：AX 文本可信度高于 OCR

- **5.2 语义理解层**
  - **5.2.1 VLM 语义摘要生成**
    - 端侧多模态大模型选择（qwen3-vl 等）与量化部署
    - 提示词设计：输入（截图 + 提取文本 + 元数据），输出（场景叙述、内容概括、主题标签）
    - 异步生成策略：独立 Description Worker，失败可重试
  - **5.2.2 摘要质量评估与一致性校验**
    - 描述与截图内容的一致性校验
    - 对后续检索召回和推理质量的贡献评估

- **5.3 多源内容融合**
  - 融合对象：AX 文本、OCR 文本、VLM 语义描述
  - 融合策略：拼接为统一全文索引（frames_fts），纳入元数据
  - 来源标注与质量分级：AX（r=1）> 混合（r=0.5）> OCR（r=0）
  - 结构化替代纯视觉存储：VLM 描述作为该帧内容的最高质量自然语言概括

**篇幅预估**：6-7页（约3000-3500字）

### 第6章 记忆存储与生命周期管理

**目标**：回答"如何将第5章生成的多源内容持久化存储并管理其生命周期？"

- **6.1 记忆单元数据模型**
  - 帧主表（frames）：核心元数据（时间戳、应用、窗口、URL、触发类型、截图路径等）
  - 文本从表：OCR 结果、AX 结果
  - 摘要表：AI 生成的结构化描述（场景叙述、内容概括、主题标签）——对应第5章 5.2 的输出
  - 元素表：结构化界面信息（角色、文本、坐标、层级）
  - 关系设计与查询按需连接

- **6.2 多阶段异步处理流水线**
  - 三阶段关系：第5章完成文本提取与语义摘要 → 本阶段负责状态管理与阶段衔接
  - 独立状态字段追踪每阶段完成情况（文本提取 / 语义摘要 / 向量嵌入）
  - 阶段间相互独立、异步推进，失败可独立重试
  - 所有阶段成功后，visibility_status 由 "pending" 变为 "queryable"

- **6.3 存储压缩与生命周期管理**
  - 采集端去重：内容哈希 + 感知哈希拦截重复帧
  - 结构化替代：优先提取 AX 结构化文本替代纯视觉存储
  - 主动压缩：基于重要性评估和访问频率，对老旧记忆保留高价值文本、压缩冗余

### 第7章 混合语义检索

**目标**：回答"如何从海量记忆中筛选最相关的候选帧？"

- **7.1 全文检索引擎**
  - FTS5 全文检索：单表 frames_fts 索引 full_text + 元数据
  - 概率排序函数（BM25）：综合考虑词频、文档长度、词项稀有度
  - 元数据树形索引：时间戳、应用名称、窗口标题等字段的精确过滤

- **7.2 向量语义索引**
  - 多模态嵌入模型：qwen3-vl-embedding，图像 + 文本融合为单一语义向量
  - LanceDB 本地向量数据库
  - 近似最近邻搜索（ANN），余弦距离度量语义相似度

- **7.3 倒数秩融合策略（RRF）**
  - 公式：$S_{RRF} = \frac{w_{fts}}{k + r_{fts}} + \frac{w_{vec}}{k + r_{vec}}$
  - 平滑参数 $k$、权重系数 $w_{fts}$ / $w_{vec}$
  - 优势：仅依赖排名避免评分尺度不统一，高排名结果权重更大

- **7.4 元数据多维过滤与证据定位**
  - 过滤条件：时间范围、应用名称、窗口标题、浏览器地址、焦点状态
  - 数据库层直接筛选，不参与排序计算
  - 证据定位：帧标识 → 时间戳 → 原始截图路径的一一映射
  - 结果呈现：结构化返回，多维度评分信息增强可解释性；检索结果可通过时间轴视图定位记忆发生时刻，也可通过帧卡片查看原始视觉帧以核验结果

**篇幅预估**：6-7页（约3000-3500字）

### 第8章 渐进式记忆推理（核心章）

**目标**：回答"如何用有限上下文处理海量记忆？"，并阐述 Skills 规范作为统一接口如何同时支撑内置助手的推理与外部 Agent 的集成。

- **8.1 回忆推理框架**
  - 时区感知提示词注入：当前本地日期、时区名称、UTC 偏移量
  - 查询意图理解：时间语义解析、活动类型推断
  - 检索编排：根据意图选择检索策略

- **8.2 Skills 驱动的渐进式信息披露策略**
  - 核心思想：信息获取量与查询需求精准匹配，避免过度检索
  - 四级递进：
    1. **活动概览**：宏观统计与场景描述，数据量小、信息密度高
    2. **混合检索**：执行 FTS5 + 向量双轨检索，返回相关记忆帧
    3. **帧级上下文**：高相关度帧的详细上下文（完整文本、AI 描述、URL 等）
    4. **原始视觉帧**：截图内容描述，仅在必要时触发
  - 决策树形式化：每步由大模型根据已有信息自主判断是否深入
  - 上下文占用随查询复杂度线性增长，而非恒定满载

- **8.3 上下文窗口保护策略**
  - 接口级截断：活动摘要条目数量控制、搜索结果数量限制
  - 响应级截断：超过约 5KB 的响应执行头部截断
  - 按需加载：默认搜索不返回文本内容，仅返回元数据和评分
  - 形式化描述：
    - 未经保护：$L = \sum_{i=1}^{N} l_i$
    - 截断后：$L' = \sum_{i=1}^{N} \min(l_i, L_{\max})$
    - 窗口约束：$L' \leq C_{window}$，超出时优先保留排名靠前的帧

- **8.4 语义输入组装与证据选择**
  - 来源质量区分：AX（$r=1$）> 混合（$r=0.5$）> OCR（$r=0$）
  - 多维度评分展示：融合评分、全文评分、向量相似度及其排名
  - 时间序列组织：按时间戳升序排列，维护记忆时序逻辑
  - 证据选择函数：
    - 排序：$S_i = \frac{w_{fts}}{k + \rho_i^{fts}} + \frac{w_{vec}}{k + \rho_i^{vec}}$
    - 格式化：$\phi(f_i) = \langle t_i; a_i; r_i; \text{truncate}(f_i^{\text{text}}, l_{\max}); S_i \rangle$
    - 组装：$E_{text{context}} = \bigoplus_{f_i \in \mathcal{F}_{sel}} \phi(f_i)$，约束 $\sum |\phi(f_i)| \leq C_{window}$
  - 推理质量控制：基于事实回答、引用证据、冲突时优先 AX、不足时明确告知

- **8.5 MyRecall 与外部 Agent 框架的集成**
  - Skills 规范设计：结构化文档描述外部工具规范，定义记忆检索接口的调用方法、参数格式、返回语义
  - OpenClaw 集成实例：已验证的接入实践，展示调用流程与效果
  - Claude Code、Hermes Agent 等框架的接入可行性分析：
    - 架构兼容性论证（REST API、JSON 协议、标准输入输出）
    - 与 MCP（Model Context Protocol）的兼容性讨论
  - **定位总结**：Skills 规范是 MyRecall 的统一记忆访问接口。内置的个人回忆助手与外部 Agent 框架（如 OpenClaw）**基于同一套接口调用记忆服务**，在架构上完全平级。MyRecall 并非传统意义上的"应用+插件"模式，而是以一个统一记忆内核支撑多种交互形态：既作为用户直接使用的个人回忆助手，又作为面向 Agent 框架的记忆服务提供者。

**篇幅预估**：12-15页（约6000-7500字），全文最厚重章节

### 第9章 个人回忆助手的交互实现

**目标**：阐述 MyRecall 作为个人回忆助手的交互层实现，明确其与外部 Agent 的平级关系。

- **9.1 多模型集成与流式对话**
  - 主进程 - 智能体子进程分离架构
  - SSE 流式对话接口（文本增量、思考增量、对话结束事件）
  - 本地 / 远程模型切换

- **9.2 会话管理**
  - 多会话并行、独立文件存储、原子写入
  - 会话标题自动生成、CRUD 操作、会话隔离

- **9.3 交互方式概述**
  - 系统提供搜索视图（关键词/语义/混合检索）、时间轴视图、对话界面等多种交互方式
  - 搜索与时间轴视图为第7章混合检索结果的直接呈现，对话界面为第8章渐进式推理的直接入口
  - 上述交互方式均基于同一 Skills 接口获取记忆数据

- **9.4 定位说明**
  - 本层是 MyRecall **作为个人回忆助手**的直接呈现形态
  - 与外部 Agent 框架（OpenClaw、Claude Code 等）**平级**
  - 均通过同一 Skills 接口调用记忆服务，不存在架构特权
  - 内置助手的对话推理与外部 Agent 的工具调用，在记忆访问层面完全一致
  - 设计目的：提供开箱即用的交互体验，而非排他性的封闭生态

**篇幅预估**：3-4页（约1500-2000字），简要描述

### 第10章 实验与评估

**目标**：设计实验方案，定义评估指标，预期对比实验结果。

> 注：本章目前为实验方案设计，具体数据需后续补充测试。

- **10.1 实验环境与数据集**
  - 硬件环境：端侧设备配置
  - 软件环境：模型版本、依赖库版本
  - 测试数据集：日常使用场景模拟、截图规模

- **10.2 评估指标**
  - 信息采集：AX 文本提取准确率、OCR 识别准确率、去重效率（冗余降低率）
  - 检索性能：召回率@K（K=5, 10, 20）、精确率、响应延迟（P50, P95, P99）
  - 推理质量：回答相关性（人工评分）、上下文利用率、幻觉率
  - 资源占用：CPU 使用率、内存占用、磁盘增长速率

- **10.3 对比实验设计**
  - 检索模式对比：全文 vs 向量 vs 混合（RRF），各取不同权重组合
  - 渐进式披露 vs 一次性注入对比：相同查询下上下文占用、回答质量、响应延迟
  - 基线对比：与纯关键词检索、纯向量检索的效果对比

- **10.4 实验结果与分析（预期）**
  - 各指标预期趋势分析
  - 参数敏感性分析（RRF 权重、平滑参数 $k$、截断阈值等）

- **10.5 与现有系统的定性对比**
  - 功能对比表：开放 API、Agent 集成、隐私策略、渐进式推理等维度
  - MyRecall 相对于 Rewind / Recall 的差异化优势

**篇幅预估**：5-6页（约2500-3000字），含数据后扩充至 8-10 页

### 结论

- **工作总结**：回顾各章核心内容
- **主要创新点**：
  1. Host-Edge 分离的端侧多模态记忆采集与存储架构
  2. FTS5 + 向量双轨混合检索与 RRF 融合策略
  3. 基于 Skills 的渐进式记忆推理方法（核心）
  4. 面向 Agent 的个人记忆服务架构
- **不足与展望**：
  - 长时运行稳定性验证
  - 更多 Agent 框架的深入集成
  - 跨设备记忆同步
  - 记忆压缩算法的进一步优化

## 5. 写作注意事项

### 5.1 与中期报告的关系

- 中期报告是工程进度汇报，毕业论文是学术成果总结
- 中期报告中的技术细节（如具体的代码实现、配置参数）在论文中应提炼为设计决策与原理阐述
- 中期报告中的图表可复用，但需重新标注为论文格式（图X.Y，表X.Y）
- 所有内容需重新撰写，不能照搬

### 5.2 与开题报告的关系

- 开题报告的研究方案已在系统中实现，论文需展示实现结果而非仅方案设计
- 开题中的预期目标需在论文中给出实际达成情况的说明
- 研究内容需深化，不能停留在开题层面的概括描述

### 5.3 格式规范

- 严格遵循大连理工大学本科毕业论文模板
- 章节编号：阿拉伯数字，如"第1章"、"1.1"、"1.1.1"
- 图/表编号：分章编号，如"图3.1"、"表7.2"
- 公式编号：分章编号，如"(8.1)"
- 参考文献：按引用顺序编码，不少于10篇（含外文期刊）
- 术语一致性：全文统一使用"Host"、"Edge"、"AX"、"OCR"、"RRF"等缩写

### 5.4 数据补充计划

| 实验项 | 当前状态 | 补充计划 |
|--------|---------|---------|
| 信息采集准确率 | 功能验证，无定量数据 | 设计测试集，统计 AX/OCR 准确率 |
| 检索召回率@K | 功能验证，无定量数据 | 构造查询-答案对，评估召回率 |
| 响应延迟 | 功能验证，无定量数据 | 压力测试，统计 P50/P95/P99 |
